# Contributing to SafeChat

First off, thank you for considering contributing to SafeChat! 

SafeChat is an AI-powered social platform designed to block cyberbullying *before* it reaches the victim. We welcome community contributions, but because of the sensitive nature of user data and the strict requirements around our moderation engine, we have specific guidelines you must follow.

## Phase-Aware Development

SafeChat is being built in phases. Please review [`docs/ROADMAP.md`](ROADMAP.md) before starting any work.

**Rule #1: Do not build out of sequence.**
If you submit a PR for a feature that belongs in Phase 6 while we are currently in Phase 1, it will be rejected. Premature features bloat the codebase and block critical path work.

## Development Setup

### 1. Backend (Python / FastAPI)
The backend requires Python 3.11+.

```bash
cd backend
python -m venv venv
# Windows: venv\Scripts\activate
# Mac/Linux: source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env # Fill in your local Firebase/GCP credentials
uvicorn main:app --reload
```

### 2. Frontend (Flutter 3.x)
The frontend is a Flutter application targeting Android and Web.

```bash
cd frontend
flutter pub get
cp .env.example .env # Fill in your local Firebase web credentials
flutter run -d chrome # Or android
```

## The Golden Rule: Never Bypass Moderation

Every endpoint that accepts user-generated content (messages, posts, comments, stories, bios) MUST route that content through the moderation engine before persisting it to the database.

- **No direct writes:** The Flutter client must *never* write user content directly to Firestore.
- **Backend enforcement:** The FastAPI backend is responsible for calling the moderation engine and returning a `422` error if the content is toxic.

## Code Conventions

### Flutter (Frontend)
- **Formatting:** We strictly use `dart format` (line length 100) and `flutter_lints`.
- **Architecture:** We use Riverpod for state management and GoRouter for declarative routing.
- **Design:** Ensure your UI components support our design system (Material 3) as outlined in [`UI_DESIGN_SYSTEM.md`](UI_DESIGN_SYSTEM.md).

### Python (Backend)
- **Formatting:** We strictly use `Black` (line length 100) and `Ruff` for linting.
- **Typing:** `mypy` strict mode is required. All functions must have type hints.
- **Pydantic:** All request and response shapes must be validated via Pydantic models.

## Testing Requirements

Your pull request will not be merged unless it includes appropriate tests:

- **Backend:** Write `pytest` tests for any new API routes. Mock the Firebase Admin SDK and Moderation APIs.
- **Frontend:** Write Flutter Widget tests for new UI components.
- **Do not write tests for trivial wrappers or third-party SDKs.** Focus on business logic and moderation flows.

## Pull Request Process

1. **Open an Issue First:** For any significant architectural change or new feature, open an issue to discuss it before writing code.
2. **Branching:** Use descriptive branch names (e.g., `feature/profile-editing`, `fix/chat-listener`).
3. **Draft PRs:** If your PR isn't ready for review, open it as a Draft.
4. **Documentation:** If you change an API endpoint, database schema, or architecture, you MUST update the relevant markdown files in `docs/` in the same PR.
5. **Clean Commits:** Write conventional commit messages.

## AI Code Assistants

If you use AI assistants (Claude, Gemini, Copilot) to generate code:
- Ensure the AI reads `AGENT.md` first.
- Do not leave "Generated by AI" footers in your commit messages or source files.
- You are strictly responsible for the code you submit. The AI is a tool, not the author.

Thank you for helping make social media safer!
